Explain what the code/change we just wrote or discussed actually does, in plain
language, as if teaching someone who is building computational skills from
scratch and wants to genuinely understand it — not just get output.

Structure it as:

**What this does (plain language):**
- [1-3 sentences, no jargon left unexplained. If a technical term is necessary,
  define it briefly inline the first time it's used.]

**Why it's built this way:**
- [1-3 sentences on the reasoning — why this approach, this tool, this order of
  operations, rather than an alternative.]

**Key part to understand:**
- [Point to the one or two lines/steps that matter most conceptually — not a
  line-by-line walkthrough, just the part that, if understood, means the rest
  makes sense.]

**If this breaks, look here first:**
- [1-2 likely failure points and what a failure would look like, so future
  debugging isn't starting from zero.]

Rules:
- Assume general scientific literacy but not programming background — explain
  code concepts (loops, API calls, parsing, etc.) the way you'd explain a new
  lab technique, not the way a CS textbook would.
- Do not oversimplify to the point of being inaccurate — the goal is real
  understanding, not reassurance.
- Keep it concise. This is a learning aid, not a lecture.
- Do not just restate the code in English line by line — explain the logic and
  purpose, not the syntax.
